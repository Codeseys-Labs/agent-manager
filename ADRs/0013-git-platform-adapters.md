# ADR-0013: Git Platform Adapters

## Context

agent-manager uses git as its sync transport (ADR-0002). While the core git
operations (init, commit, push, pull) work with any remote, git hosting platforms
offer additional features that agent-manager could leverage:

- **GitHub:** Secrets API, Actions CI, Pull Request creation, repository creation
- **GitLab:** CI/CD Variables, Pipelines, Merge Request creation
- **Gitea/Forgejo:** Secrets API (limited), Actions (Gitea 1.19+)
- **Bare git (SSH/local):** No platform features, just git transport

The encryption model (ADR-0012) needs a way to optionally store decryption keys in
platform secret stores. The `am clone` flow needs to work with any remote. Future
features (auto-create repo, CI config generation, PR-based config proposals) need
platform awareness.

This parallels how IDE adapters translate between core config and native formats —
git platform adapters translate between core git operations and platform-specific APIs.

## Decision

Introduce a second adapter dimension: **git platform adapters**, alongside the
existing IDE adapters.

### Two adapter axes

```
IDE Adapters (what tool to configure):
  claude-code, cursor, windsurf, copilot, codex, kilo, kiro, forgecode, ...

Git Platform Adapters (where configs are stored):
  github, gitlab, gitea, bare
```

### Platform adapter interface

```typescript
interface GitPlatformAdapter {
  meta: {
    name: string;           // "github"
    displayName: string;    // "GitHub"
  };

  // Detection
  detect(remoteUrl: string): boolean;  // Does this URL belong to this platform?

  // Authentication
  login(): Promise<AuthResult>;        // OAuth device flow or token prompt
  isAuthenticated(): Promise<boolean>;

  // Key management (ADR-0012 integration)
  storeKey?(key: string): Promise<void>;   // Store encryption key in platform secrets
  retrieveKey?(): Promise<string | null>;  // Retrieve key from platform secrets

  // Platform features (optional, Phase 2+)
  createRepo?(name: string, options: RepoOptions): Promise<string>;  // Returns URL
  createPR?(title: string, body: string): Promise<string>;           // Returns PR URL
}
```

### Built-in platform adapters

| Adapter | Detection | Auth | Key Storage | Phase |
|---------|-----------|------|-------------|-------|
| `bare` | Default fallback | SSH keys / HTTP credentials | Local file only | 1 |
| `github` | URL contains `github.com` | `gh auth` / OAuth device flow | GitHub Secrets API | 1 |
| `gitlab` | URL contains `gitlab` | `glab auth` / OAuth | GitLab CI/CD Variables API | 2 |
| `gitea` | Configurable URL pattern | Token-based | Gitea Secrets API | 2 |
| `codeberg` | URL contains `codeberg.org` | Token-based | Codeberg Secrets API (Forgejo) | 3 |

### Detection logic

```typescript
function detectPlatform(remoteUrl: string): GitPlatformAdapter {
  if (remoteUrl.includes("github.com")) return githubAdapter;
  if (remoteUrl.includes("gitlab"))     return gitlabAdapter;
  // Check for gitea/forgejo by API probing (Phase 2)
  return bareAdapter;  // Default: no platform features
}
```

### How it integrates with encryption (ADR-0012)

```bash
# First machine — GitHub remote
am init
# → Generates encryption key
# → Detects GitHub remote
# → "Store encryption key in GitHub Secrets? [Y/n]" y
# → Calls GitHub Secrets API: sets AM_ENCRYPTION_KEY on the repo

# Second machine — clone
am clone git@github.com:user/config.git
# → Detects GitHub remote
# → "Retrieve encryption key from GitHub Secrets? [Y/n]" y
# → Calls GitHub Secrets API... but wait, Secrets are write-only!
```

**GitHub Secrets limitation:** The GitHub Secrets API is write-only — you can
create/update secrets but cannot read values back. This means:

- **CI/CD:** Works. GitHub Actions can read secrets as env vars.
- **Local dev:** Does NOT work for reading the key back.

**Resolution:** Platform adapters handle key storage as a **convenience for CI/CD**,
not as the primary key distribution mechanism. For local dev, the key lives in
`.agent-manager/key.txt` and is shared via password manager.

**Key distribution matrix:**

| Context | Key source |
|---------|-----------|
| First machine | Generated by `am init` |
| Other machines | Pasted from password manager |
| CI/CD (GitHub) | `${{ secrets.AM_ENCRYPTION_KEY }}` |
| CI/CD (GitLab) | `$AM_ENCRYPTION_KEY` (CI variable) |
| CI/CD (Gitea) | `$AM_ENCRYPTION_KEY` (secret) |
| CI/CD (bare) | `$AM_ENCRYPTION_KEY` (env var) |

### The `bare` adapter

The `bare` adapter is the default and works with any git remote:

```typescript
const bareAdapter: GitPlatformAdapter = {
  meta: { name: "bare", displayName: "Git" },
  detect: () => true,  // fallback, matches everything
  login: async () => ({ authenticated: true }),  // assumes SSH/HTTP auth works
  isAuthenticated: async () => true,
  // No storeKey/retrieveKey — local key file only
};
```

This ensures agent-manager works with Gitea, Forgejo, Codeberg, Gogs, BitBucket,
Azure DevOps, AWS CodeCommit, or any bare SSH repo without any platform-specific code.

### Future platform adapters (Phase 2+)

The following platforms are candidates for dedicated adapters when demand warrants:

| Platform | Based On | Detection | Notes |
|----------|----------|-----------|-------|
| **Gitea** | Gitea | API probing (`/api/v1/version`) | Self-hosted, Gitea Actions (1.19+) |
| **Codeberg** | Forgejo | `codeberg.org` in URL | Largest Forgejo instance, same API as Gitea |
| **Forgejo** | Forgejo | API probing | Fork of Gitea, API-compatible |
| **BitBucket** | Atlassian | `bitbucket.org` in URL | Pipelines variables for CI key storage |

All of these are covered by the `bare` adapter today. Dedicated adapters would add:
CI key storage via platform secrets, repo creation, and PR/MR creation.
The Gitea/Forgejo/Codeberg adapters can share a common base since their APIs are
compatible — a single `gitea-compatible` adapter with URL-based sub-detection.

## Consequences

### Positive
- Works with ANY git remote by default (bare adapter)
- Platform-specific features are opt-in, not required
- CI/CD key distribution is automated for GitHub/GitLab
- Follows the same adapter pattern as IDE adapters — consistent architecture
- New platforms can be added without core changes

### Negative
- Two adapter dimensions (IDE + platform) adds conceptual complexity
  (mitigation: most users never think about platform adapters — auto-detected)
- GitHub Secrets API is write-only — can't retrieve keys for local dev
  (mitigation: password manager for local, platform secrets for CI only)

### Neutral
- Platform adapter detection is URL-based — simple regex, no API probing needed
- The `gh` and `glab` CLIs can be used as auth helpers if installed

## Alternatives Considered

- **No platform adapters (git-only):** Rejected — loses CI/CD key distribution,
  repo creation convenience, and future PR-based config proposals.
- **GitHub/GitLab-only (no bare):** Rejected — excludes self-hosted users,
  contradicts the self-hostable principle.
- **Platform features in core (not adapters):** Rejected — would hardcode GitHub/GitLab
  assumptions into the core engine.
