# Lens B: ADR-0043 hosted UI auth implementation research

Date: 2026-05-05
Scope: Cloudflare Worker hosted UI auth, multi-git-backend abstraction, stateless encrypted sessions, and per-MCP secret references for ADR-0043.

## Findings

ADR-0043's five-tier matrix is directionally aligned with current production practice, but implementation should narrow the abstraction from "OAuth provider" to "repo capability route". Cloudflare Workers are a good fit for redirect/callback OAuth and token-handler patterns because the runtime exposes Web Crypto, secret bindings, fetch, and standard Request/Response APIs. Cloudflare's 2026 Worker best-practices docs emphasize Web Crypto for secure random values, no mutable global state assumptions, current compatibility dates, and bindings/secrets over ad-hoc external calls. Their security model also assumes isolates are preemptible and request-scoped, so session state must be either carried by the request or stored in a Cloudflare storage primitive. ADR-0015 and ADR-0043 choose the former.

For Cloudflare Workers OAuth in 2025-2026, the canonical recipe is: `/auth/:provider/login` generates a high-entropy `state`, PKCE `code_verifier`, and S256 `code_challenge`; stores `state`, provider, redirect target, and code verifier in an HttpOnly, Secure, SameSite=Strict login-state cookie; redirects to the provider's authorization URL with a provider-specific redirect URI; `/auth/:provider/callback` verifies state, exchanges the code and verifier at the token endpoint, normalizes the token response, and writes an encrypted session cookie containing only the fields needed for later API calls. Exa surfaced production examples in the Cloudflare ecosystem: `data-pond/sveltia-cms-auth` is a Worker OAuth bridge for Sveltia/Decap CMS that registers GitHub and GitLab callbacks at `/callback`, uses `api` for GitLab, supports allowed-domain restrictions, and explicitly notes PAT entry as the simpler single-user alternative. Curity's Cloudflare OAuth proxy Worker implements the token-handler pattern: decrypt a cookie, attach the access token as an Authorization header, and reject untrusted origins. Wristband's Cloudflare Workers auth SDK recommends cookie-based sessions as lighter than Redis-like stores and shows Iron Session-style HttpOnly, Secure, SameSite session cookies.

Provider behavior supports the ADR split. GitHub's OAuth docs now tell new integrations to prefer GitHub Apps over OAuth Apps where possible, because Apps have fine-grained permissions and installation tokens. For this project, GitHub.com should use a GitHub App installation flow, not generic OAuth: the cookie holds `installation_id` and selected repo identity; the Worker signs a JWT with its GitHub App private key and mints a short-lived installation token per request. GitLab supports OAuth2 applications and documents `api` as full read/write API access; ADR-0043 correctly preserves the known `write_repository` insufficiency for API commit paths. GitLab refresh tokens are rotating in modern GitLab deployments, so two parallel refreshes can invalidate one another. Gitea/Forgejo support OAuth2 provider flows, PKCE for public/mobile clients, refresh tokens, and per-instance redirect URI registration, but this is precisely why arbitrary self-hosted Gitea/Forgejo cannot be a hosted global OAuth tier: every instance is its own IdP and many admins will not enable dynamic client registration. Codeberg is special only because it is one known Forgejo instance where the hosted service can pre-register a client.

The multi-backend abstraction pattern in production tools is not a single `OAuthClient`; it is a provider/SCM object plus a credential object plus a repo descriptor. DeepWiki's read of Argo CD shows `Repository` and `RepoCreds` structs with orthogonal auth fields: username/password or PAT, SSH private key, GitHub App private key/app ID/installation ID, bearer token, TLS client certs, and cloud identities. `GetGitCreds` returns a typed credential provider which the git client maps to a transport auth method. Argo CD's ApplicationSet SCM generator separately instantiates GitHub, GitLab, Gitea, Bitbucket Server, and Bitbucket Cloud providers for repository discovery. Coder has an `EnhancedExternalAuthProvider` family for GitHub, GitLab, Bitbucket, and Gitea defaults. Argo GitOps Promoter goes further in the ADR-0043 direction: it models `ScmProvider` and `GitRepository` separately and supports GitHub App config, GitLab PATs, Gitea PATs, Forgejo/Codeberg, Bitbucket Cloud, and Azure DevOps. The de-facto interface is therefore: detect host/capability, normalize `RepoRef`, obtain `Credential`, then expose operations such as list repos, read file, write file/commit, create branch, and open PR using the provider's native API when available.

For JavaScript libraries, `openid-client` is excellent for OIDC/OAuth metadata and standards-heavy IdPs but is not a natural fit for GitHub App installation tokens or Gitea REST quirks, and edge runtimes can make Node-oriented dependencies awkward unless they are WebCrypto/fetch-native. `simple-oauth2` is straightforward for confidential OAuth clients but historically assumes Node conventions and does not solve provider API normalization. A small in-house OAuth helper using Web Crypto, fetch, URLSearchParams, and typed provider descriptors is likely safer in a Worker than pulling in a large Node OAuth stack. Use libraries for JWT signing if they are WebCrypto-compatible; otherwise implement minimal RS256/PKCS8 import via `crypto.subtle` for GitHub App JWTs.

Encrypted-cookie sessions remain the best match for ADR-0015. Iron Session's core model -- sealed, signed/encrypted cookies with no server-side session IDs -- is still recommended by multiple serverless-auth examples, but the project should not depend on a Next/Node wrapper unless it is known to run cleanly in Workers. Implement a tiny Worker-native seal instead: HKDF from `SESSION_SECRET` to AES-GCM and HMAC or use an authenticated AEAD envelope only; include version, key id, issued-at, expiry, provider, tier, token material, and a refresh nonce. Cookie attributes should be `HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=<short>`. The cookie threat model is simple: an attacker who steals the browser cookie can act as the user until expiry or revocation; HttpOnly mitigates casual XSS token reads but not CSRF on state-changing endpoints unless SameSite, origin checks, and anti-CSRF headers are enforced; compromise of `SESSION_SECRET` decrypts every active session cookie, so dual-key rotation and short max age are required. KV-backed sessions or Durable Objects improve revocation and refresh serialization but violate the ADR's no per-user server state. If ADR-0043 keeps strict statelessness, accept weak revocation and solve races with optimistic cookie replacement and retry-once behavior.

Per-MCP secret URI conventions are fragmented. Environment expansion is the broadest common denominator: Gemini CLI documents `$VAR`, `${VAR}`, and Windows `%VAR%` expansion in MCP server `env`, and LibreChat supports `${ENV_VAR}` in MCP headers plus `{{USER_FIELD}}` substitutions and OAuth2/DCR for remote MCP servers. 1Password uses `op://vault/item/field` secret references with `op run`/`op inject`; Claude Code users are actively requesting native `op://` support in MCP settings. Helm secrets/jkroepke-style tooling uses explicit provider URI schemes such as `ref+vault://`, `ref+sops://`, `ref+keychain://`, `ref+k8s://`, and `secretref+k8s://`. Security guidance ranks OS keychain, environment variables, managed secret services such as Vault/AWS Secrets Manager/Azure Key Vault, and OAuth flows as preferable to plaintext MCP config. There is not yet a standard `keychain://` or `vault://` MCP secret URI, so agent-manager should treat these as resolver plugins, not as interoperable standards.

## Recommended architecture

Implement ADR-0043 as a route/capability dispatcher rather than an OAuth-provider registry. Define:

```ts
type AuthTier = 'github-app' | 'oauth-pkce' | 'gitea-pat' | 'iso-git' | 'ssh-blocked';
interface RepoRoute { tier: AuthTier; provider: string; repo: RepoRef; capabilities: CapabilitySet; }
interface GitBackend {
  detect(url: URL): Promise<Detection>;
  beginAuth?(ctx): Promise<Response>;
  finishAuth?(ctx): Promise<SessionPatch>;
  refresh?(session): Promise<SessionPatch>;
  readFile(route, path, session): Promise<FileBlob>;
  writeFile(route, path, content, message, session): Promise<CommitRef>;
}
```

The dispatcher should first reject SSH forms before `URL()` parsing, then match GitHub.com, GitLab.com or detected GitLab, Bitbucket.org, Codeberg.org, probed Gitea/Forgejo via `/api/v1/version`, generic HTTPS smart-HTTP, and finally blocked/unsupported. Keep ADR-0025's provider registry idea internally, but do not expose it as the primary architecture; the real branch is by auth tier and read/write API capability.

For GitHub, store only `{tier:'github-app', installation_id, owner, repo, created, expires_at}` in the encrypted cookie. Mint installation tokens per request and cache only within the request. For GitLab, Bitbucket, and Codeberg, use OAuth2 authorization-code + PKCE. The login-state cookie holds `state` and `code_verifier`; the session cookie holds `access_token`, `refresh_token`, `expires_at`, `provider`, `tier`, `repo`, and `refresh_nonce`. Refresh when less than 60 seconds remain; if a 401 occurs, retry once after refreshing. Because refresh rotation races are unsolved without server state, include the nonce and return the freshest cookie on every refresh response; concurrent losers should force a clean re-login rather than silently corrupting the session.

For self-hosted Gitea/Forgejo and generic HTTPS git, keep PATs out of Worker cookies. Browser stores PATs in IndexedDB encrypted with WebCrypto using a passphrase-derived non-extractable key. Requests that need the PAT send it over HTTPS to the Worker/API target only for that operation. Prefer direct REST to Gitea/Forgejo when CORS allows; use the Worker as a narrowly scoped authenticated proxy only when the instance lacks CORS. For generic HTTPS git, use isomorphic-git over OPFS and an authenticated, origin-limited CORS proxy; never ship an open proxy.

For MCP secrets, support a resolver contract in config, not raw expansion everywhere: `${VAR}` and `env://NAME` for environment variables; `op://...` via 1Password CLI if installed; `keychain://service/account` for local CLI only; `vault://path#field` for HashiCorp Vault; and later `aws-secrets://`, `gcp-secret://`, `azure-keyvault://`, or `ref+sops://`. Hosted UI should display unresolved references and allow test-resolution only when the resolver is browser-safe or proxied by an explicitly authenticated remote MCP server. Do not let the Worker resolve local keychain or local env references.

## Tradeoffs

Encrypted cookies preserve statelessness, horizontal scalability, and low latency, but they sacrifice server-side revocation, precise audit trails, and refresh serialization. KV or Durable Objects would solve refresh races and logout revocation but create per-user server state and change ADR-0015's operating model. A compromise is optional deployment mode: hosted public service remains stateless; self-hosted enterprise deployments may enable a KV session backend with the same `SessionStore` interface.

A custom Worker-native OAuth helper increases maintenance burden but avoids dragging Node-specific OAuth libraries into an edge runtime and keeps provider quirks explicit. `openid-client` is better if the product later needs standards-first OIDC discovery for remote MCP servers; it should not be the foundation for git provider auth.

GitHub App primary auth adds setup friction and requires app-private-key handling, but it gives the strongest least-privilege story: per-repo install and one-hour installation tokens. PAT fallback everywhere would be much faster to implement, but it permanently damages trust in the hosted UI.

PATs in IndexedDB are a UX/security compromise. They avoid storing self-hosted secrets in Worker cookies and support devices without server state, but XSS in the hosted UI can still ask the browser to decrypt and exfiltrate active credentials. This makes CSP, bundle integrity, dependency pinning, and short unlock windows non-negotiable.

## Risks

The largest production risk is GitLab/Codeberg refresh-token rotation under concurrent Worker requests. Without a server lock, occasional forced re-login is likely. Test this early with parallel saves and background repo refreshes.

The second risk is CORS proxy abuse. Any Tier 4 proxy must require an authenticated session, restrict target schemes to HTTPS, block private IP ranges and metadata endpoints, enforce origin allowlists, rate limit by session and IP, and log target host aggregates without logging tokens.

The third risk is cookie compromise. HttpOnly cookies reduce script access but do not protect against browser/profile theft, malicious extensions, or `SESSION_SECRET` leakage. Use short cookie lifetimes, key ids, dual-key rotation, provider token revocation links in logout UX, and no sensitive values in logs.

The fourth risk is false support claims for self-hosted OAuth. Gitea/Forgejo OAuth works per instance, but a public hosted UI cannot assume client registration. ADR-0043 should continue to present PAT entry as the honest default for Tier 3, with OAuth only for configured known instances.

Finally, MCP secret URI support can become a portability trap if agent-manager invents schemes and implies ecosystem support. Mark `${VAR}`/`env://` as portable, `op://` as de-facto 1Password, and `keychain://`/`vault://` as agent-manager resolver plugins with explicit host/runtime limits.
